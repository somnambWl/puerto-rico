# 06 — UI (Backend + Frontend)

## Purpose

Let a human play a full game against any agent (heuristic first, RL later) in a browser. The UI is a
thin client over an authoritative server-side engine; it reuses `state.legal_actions()` for move
legality and never reimplements rules.

```
puerto_rico/ui/
├── backend/        fastapi app: game session manager, websocket protocol, agent host
│   ├── app.py          HTTP + WebSocket routes
│   ├── session.py      GameSession = one Game + AI agent + seat assignment
│   ├── schemas.py      Pydantic models for wire protocol
│   ├── catalog.py      static building catalog + good base values
│   └── labels.py       action → human-readable label (e.g. "Ship 3 sugar to ship #1")
└── frontend/       React + TypeScript + Vite; renders board, prompts legal moves
```

## Backend

**Responsibilities**
- Own the authoritative `Game` (engine). The browser never holds rules; it sends a chosen action id
  and receives the new public view.
- **Decide everything.** The server pre-computes every value the UI displays — whose turn it is, the
  legal actions and their human-readable labels, already-discounted building costs, production results,
  VP breakdowns, and per-action highlight hints. The frontend renders these as-is and derives nothing.
- Host the opponent agent (HeuristicAgent or the loaded RL policy via `rl_policy.py`). When it is an
  AI seat's turn, the server steps the agent automatically and streams the resulting states.
- Expose the **same atomic-decision contract** the engine uses: each human decision = pick one of the
  currently legal actions.

**Session model (session.py)**
```python
class GameSession:
    game: Game
    human_seat: int
    ai: Agent                              # one policy drives ALL non-human seats (3 in a 4-player game)
    def state_view(self) -> dict           # engine.public_view(perspective=human_seat) + legal-action list
    def human_step(self, action_int) -> list[dict]   # apply human action, then auto-run AI until human's turn or terminal; return the sequence of views for animation
```

**HTTP/WebSocket protocol (schemas.py)**

- `POST /games` → `{seed?, human_seat?, opponent: "heuristic"|"rl", difficulty?}` ⟹ `{game_id, state}`.
  Creates a new game and initializes it; if the human is not first, runs AI until the human's turn.

- `WS /ws/games/{game_id}` — persistent WebSocket connection for the game session:
  - **Server → Client `StateMsg`:** `{view, legal_actions: [{id, label, kind, detail}], to_move,
    terminal, winner?, preview?: bool}`. Legal actions come from `ActionCodec` + human-readable labels
    from `labels.py` (e.g. "Build Coffee Roaster (cost 6)", "Ship 3 sugar to ship #2", "Take Indigo
    plantation"). `to_move` is the seat index; `preview` marks a hypothetical frame (from E5).
  - **Client → Server `ActionMsg`:** `{action_id}` (must be one of the currently-legal ids; server
    validates).
  - **Server → Client `SequenceMsg`:** ordered list of `StateMsg` views covering the AI's intermediate
    decisions (one per AI action), so the frontend can animate the opponent's turn step-by-step.

- `GET /games/{game_id}` → current `StateMsg` (reconnect/refresh).

- `POST /games/{game_id}/preview` (E5) — preview endpoint for hypothetical play:
  Request body: `{action_id}`. Applies the action to a **cloned** game (no validation, no AI), returns
  the new `StateMsg` with `preview: true`. The real game is never mutated. Client uses this to show
  "what happens if I do this?" tooltips before the human commits. Response: `200 StateMsg`; `404` game
  not found; `400` missing/invalid action_id.

- `GET /catalog` (E5) → static building catalog + good reference:
  ```json
  {
    "buildings": [
      {
        "id": 0,
        "name": "Small Market",
        "cost": 1,
        "column": "small",
        "vp": 1,
        "capacity": null,
        "is_large": false,
        "is_production": false,
        "produces": null,
        "kind": "small",
        "description": "Player may buy goods at discounted price..."
      },
      ...
    ],
    "goods": [
      {"good": 0, "name": "Corn", "base_value": 0},
      {"good": 1, "name": "Indigo", "base_value": 3},
      ...
    ]
  }
  ```
  Used by the UI for tooltips, the VP-sorted build shelf (E6), and good-value displays.

**Labels.** The server is responsible for turning action ids into friendly labels (it has the engine
state). Keep labeling in one module so it's consistent and testable.

## Frontend

**Stack:** React + TypeScript + Vite. No game logic in the client; it renders `view` and the offered
`legal_actions`, and posts the chosen `action_id`.

**Views/components**

- **Board:** the shared board — role placards (accumulated doubloons; highlight available ones in
  role-selection), colonist ship, cargo ships (good + fill), trading house, supplies, face-up
  plantation row, VP-chip pool, **buildings-available shelf (E6: VP-sorted)**.
- **PlayerBoard ×4:** island (12 slots + colonist indicators), city (12 slots, large buildings span
  two), doubloons, goods (with trading values from `/catalog`, E6), stored colonists. Opponents' VP
  hidden (shown as unknown). Human's board prominent; AI boards compact.
- **Action prompt (E6: enhanced):** context panel listing legal actions as buttons from
  `legal_actions[].label`. Tapping sends the `action_id`. Reads naturally: "Settler: choose a tile",
  "Mayor: place a colonist (4 left)", "Captain: choose a good to ship", etc. **Highlight on the board:**
  hover/focus an action → highlight its target (tile, building, ship, good). **Tooltip on preview:**
  shows the hypothetical board state via `POST /games/{id}/preview` before commit (E5).
- **Log:** running human-readable history of decisions (built from action label stream + labels.py).
  Shows who took what action in what phase.
- **Game-over:** final scores with VP breakdown (chips + building VP + large-building bonuses from
  `scoring.py`), and the winner. Rankings table.

**Enhancements (E5, E6):**
- E5: Action preview (`POST /games/{id}/preview`) — client can show "what happens if you do this?"
  before the human commits, using the hypothetical state.
- E6: Tooltips (building descriptions from `/catalog`), step/pause AI playback, VP-sorted building
  shelf, good trading values (base_value from `/catalog`).

**Interaction flow**
1. On connect, render the `StateMsg`. If `to_move == human_seat`, enable the action buttons.
2. On click, send `ActionMsg`; disable input; await `SequenceMsg`.
3. Animate the AI's sequence (step through intermediate `StateMsg`s with a short delay), then re-enable
   input when it's the human's turn or show game-over.

**Difficulty.** Map "opponent + difficulty" to: heuristic, or an RL checkpoint (optionally a weaker
earlier checkpoint for an easier game). The server selects the agent at session creation.

## Why this design

- **Authoritative engine** = no rules duplication and no cheating risk; the browser is a renderer.
- **Same `legal_actions` contract** end-to-end means the hardest part (legality) is solved once in the
  engine and reused by RL, the arena, and the UI alike.
- **Atomic-decision prompts** make a complex game approachable: the human is always answering one clear
  question with a short list of buttons.

## Acceptance criteria

- `POST /games` then a WS round-trip yields a valid initial `StateMsg` with correctly labeled legal
  actions for the human's first decision.
- Playing a full game vs the heuristic agent through the UI reaches a correct game-over screen with a
  VP breakdown matching `scoring.py`.
- Every offered `action_id` is accepted by the engine (no desync between labels and legality), verified
  by an automated client test that plays random legal actions through the WS protocol to termination.
- Swapping `opponent: "rl"` loads a checkpoint via `rl_policy.py` and plays without server-side rules
  code changing.
