# 06 — UI (Backend + Frontend)

## Purpose

Let a human play a full game against any agent (heuristic first, RL later) in a browser. The UI is a
thin client over an authoritative server-side engine; it reuses `state.legal_actions()` for move
legality and never reimplements rules.

```
puerto_rico/ui/
├── backend/        fastapi app: game session manager, websocket protocol, agent host
│   ├── app.py
│   ├── session.py      one GameSession = one Game + the AI agent + seat assignment
│   └── schemas.py      pydantic models for the wire protocol
└── frontend/       react + typescript (vite); renders board, prompts legal moves
```

## Backend

**Responsibilities**
- Own the authoritative `Game` (engine). The browser never holds rules; it sends a chosen action id
  and receives the new public view.
- **Decide everything.** The server pre-computes every value the UI displays — whose turn it is, the
  legal actions and their human-readable labels, already-discounted building costs, production results,
  VP breakdowns, and per-action highlight hints. The frontend renders these as-is and derives nothing.
- Host the opponent agent (HeuristicAgent or the loaded RL policy via `rl_policy.py`). When it is an
  AI seat's turn, the server steps the agent automatically and streams the resulting states.
- Expose the **same atomic-decision contract** the engine uses: each human decision = pick one of the
  currently legal actions.

**Session model (session.py)**
```python
class GameSession:
    game: Game
    human_seat: int
    ai: Agent                              # one policy drives ALL non-human seats (3 in a 4-player game)
    def state_view(self) -> dict           # engine.public_view(perspective=human_seat) + legal-action list
    def human_step(self, action_int) -> list[dict]   # apply human action, then auto-run AI until human's turn or terminal; return the sequence of views for animation
```

**HTTP/WebSocket protocol (schemas.py)**
- `POST /games` `{seed?, human_seat?, opponent: "heuristic"|"rl", difficulty?}` → `{game_id}`.
- `WS /games/{game_id}`:
  - server → client `StateMsg`: `{view, legal_actions: [{id, label, kind, detail}], to_move, terminal,
    result?}`. `legal_actions` come from `ActionCodec` + human-readable labels (e.g. "Build Coffee
    Roaster (cost 6)", "Ship 3 sugar to ship #2", "Take Indigo plantation"). Use the codec so the UI
    never derives legality.
  - client → server `ActionMsg`: `{action_id}` (must be one of the offered ids; server validates).
  - server → client `SequenceMsg`: ordered list of `StateMsg` views covering the AI's intermediate
    decisions, so the frontend can animate the opponent's turn step by step.
- `GET /games/{game_id}` → current `StateMsg` (reconnect/refresh).

**Labels.** The server is responsible for turning action ids into friendly labels (it has the engine
state). Keep labeling in one module so it's consistent and testable.

## Frontend

**Stack:** React + TypeScript + Vite. No game logic in the client; it renders `view` and the offered
`legal_actions`, and posts the chosen `action_id`.

**Views/components**
- *Board:* the shared board — role placards (with accumulated doubloons; highlight available ones in
  role-selection), colonist ship, cargo ships (kind + fill), trading house, supplies, the face-up
  plantation row, VP-chip pool, buildings-available shelf.
- *PlayerBoard ×4:* island (12 tile slots with colonist dots), city (12 building slots, large building
  spanning two), doubloons, goods, stored colonists; the three opponents' VP shown as hidden. (Show the
  human's board prominently; the three AI boards more compactly.)
- *Action prompt:* the heart of the UI — a context panel listing the current legal actions as buttons
  generated from `legal_actions`. Tapping one sends the `action_id`. Because the engine drives one
  atomic decision at a time, the prompt naturally reads as "Settler: choose a tile", "Mayor: place a
  colonist (4 left)", "Captain: choose a good to ship", etc. **Highlight on the board** the elements an
  action refers to (e.g. hover "Take Sugar" → highlight that face-up tile).
- *Log:* a running, human-readable history of both players' decisions (built from the label stream).
- *Game-over:* final scores with the VP breakdown from `scoring.py` (chips, building VP, large-building
  bonuses), and the winner.

**Interaction flow**
1. On connect, render the `StateMsg`. If `to_move == human_seat`, enable the action buttons.
2. On click, send `ActionMsg`; disable input; await `SequenceMsg`.
3. Animate the AI's sequence (step through intermediate `StateMsg`s with a short delay), then re-enable
   input when it's the human's turn or show game-over.

**Difficulty.** Map "opponent + difficulty" to: heuristic, or an RL checkpoint (optionally a weaker
earlier checkpoint for an easier game). The server selects the agent at session creation.

## Why this design

- **Authoritative engine** = no rules duplication and no cheating risk; the browser is a renderer.
- **Same `legal_actions` contract** end-to-end means the hardest part (legality) is solved once in the
  engine and reused by RL, the arena, and the UI alike.
- **Atomic-decision prompts** make a complex game approachable: the human is always answering one clear
  question with a short list of buttons.

## Acceptance criteria

- `POST /games` then a WS round-trip yields a valid initial `StateMsg` with correctly labeled legal
  actions for the human's first decision.
- Playing a full game vs the heuristic agent through the UI reaches a correct game-over screen with a
  VP breakdown matching `scoring.py`.
- Every offered `action_id` is accepted by the engine (no desync between labels and legality), verified
  by an automated client test that plays random legal actions through the WS protocol to termination.
- Swapping `opponent: "rl"` loads a checkpoint via `rl_policy.py` and plays without server-side rules
  code changing.
