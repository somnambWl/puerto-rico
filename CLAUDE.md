# Puerto Rico AI — Project Overview

A digital implementation of the board game **Puerto Rico** with a playable UI and a
reinforcement-learning opponent. Personal-use project (the game is published; do not distribute).

## What we are building

1. A fast, headless, deterministic **rules engine** (the simulator).
2. A **PettingZoo environment** wrapping the engine for RL.
3. **Baseline agents** (random + heuristic) for early play and benchmarking.
4. An **RL agent** trained by self-play (PPO + action masking).
5. A **UI** (web) so a human can play against any agent.

## Scope (first milestone)

- **4-player base game.** Standard structure: each player takes **one role per round**
  (a round = each player selects one role once, going clockwise from the governor). No expansions.
  The data model is N-player-capable; 4-player is the primary, tuned target and 2-player remains a
  config option. Note: 4-player is **general-sum** (not zero-sum), which shapes the RL reward — see
  `design/05`.
- Rules source of truth: `docs/puerto-rico-rules.md`.

## Tech stack

- Python 3.11+ for engine, env, agents, training.
- `pettingzoo` (AEC API) for the environment; `numpy` for encoding.
- `torch` (PyTorch) for custom multi-agent PPO self-play with action masking (no RLlib dependency).
- `fastapi` + `websockets` backend; `react` + `typescript` frontend for the UI.
- `pytest` for tests. `uv` for dependency management.

## Build order (dependencies)

Engine → Env → Baseline agents → RL training → UI. The engine is the foundation and
must be correct and fast before anything downstream is worth building. Baseline agents +
a thin UI can ship before any RL, so a human can play immediately.

## Key invariant

The engine is the single source of truth for legality. Nothing else (UI, agents) reimplements
rules — they all call `state.legal_actions()` and `state.apply(action)`.
